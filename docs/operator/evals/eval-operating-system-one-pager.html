Fabro Eval Operating System

Evals are treated like tests: every model call, workflow stage, product workflow, and eval itself has registered evidence. Promptfoo is a runner; the central truth is the normalized eval registry, index, and dashboard.

How The Loop Works

1 Register

Every eval lives in evals/registry.yaml with level, owner, artifacts, and waiver policy.

2 Run

Promptfoo, deterministic checks, or call evaluators produce normalized JSON results.

3 Collect

The indexer rejects spoofed results that do not match registry level, runner, workflow, node, or artifact pattern.

4 Gate

Missing, failed, or fallback-only blocking evals fail the operating gate.

5 Improve

Failures create new cases, better rubrics, or stronger workflow rules for the next run.

Clean Pass

Primary runner passed

The eval runner passed and the normalized result matches the registry contract. This is the only normal green state.

Blocking

Missing, failed, or fallback-only

Fallback checks can reduce risk, but they do not count as a clean pass. The dashboard should make this visible.

Waived

Explicit accepted risk

A waiver needs waiver_id, owner, reason, risk statement, review date, and compensating control.

Daily Operator Commands

npm run eval:registry

Validates registry shape, owners, artifact patterns, blocking status, and counterexamples.

npm run eval:coverage

Checks workflow-builder coverage and prevents new workflow work from shipping without eval accounting.

npm run eval:index

Builds reports/eval-index.json. A nonzero exit is expected when blocking evidence is missing.

npm run eval:dashboard

Renders reports/eval-dashboard.md, the quick view for what is pass, fail, missing, fallback-only, or waived.

npm run factory:dashboard

Renders reports/factory-dashboard.md for humans and reports/factory-health.json for Quincy.

What Quincy Must Do

  • Add a registry entry before adding or changing a model call, stage, workflow, or product-level output.
  • Wire each call through run-codex-prompt.mjs --eval-id and write normalized evidence.
  • Run registry and coverage checks before promoting a Fabro workflow.
  • Treat dashboard missing evidence as a blocker, not a note.
  • Only accept fallback when the structured waiver explains the risk and the compensating control.

Where To Look

  • evals/registry.yaml is the source of truth.
  • evals/result.schema.json defines normalized result evidence.
  • scripts/evals/collect-index.mjs builds the central index and rejects mismatches.
  • docs/operator/evals/EVAL-OPERATING-RUNBOOK.md is the operator runbook.
  • reports/eval-dashboard.md is the current dashboard snapshot.
  • reports/factory-dashboard.md is the central factory rollup.
  • reports/factory-health.json is the daily agent health API.

How To Add A New Eval

  • Choose the level: call, workflow, product, or meta.
  • Register owner, subject paths, artifact patterns, and at least one counterexample.
  • Emit normalized JSON under the Fabro run ID when available through FABRO_RUN_ID.
  • Add tests for the failure mode before trusting the eval as blocking.
  • Update the dashboard and make sure the result is present, matched, and not fallback-only.

The Team Rule

A workflow is not done because it ran. It is done when the model calls, stages, whole workflow, and eval machinery each have registered evidence. The factory should learn from every failure by turning it into a sharper eval, a better counterexample, or a stricter workflow rule.
